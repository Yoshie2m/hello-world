# Linear連携 `/loop` ポーリング運用 手順書（REQ1 案1）

対象: [change_plan.md](change_plan.md) REQ1 案1「対話セッション + `/loop` によるポーリング運用（最小構成）」を実際に運用するための手順書。

## 前提条件（開始前に必ず確認）

1. claude.ai側のコネクタ設定で **Linear MCP（claude.ai Linear）が認可済み** であること。
   - 未認可の場合、本セッションのようにLinear関連ツールが一切呼び出せない状態になる。
   - 認可後は、Claude Codeのセッションを開き直す（またはMCP再接続する）ことでツールが認識される。
2. 対象Linearプロジェクト: https://linear.app/deepworkyk/project/hello-world-9b426137e05a/overview
3. ローカルのこのリポジトリ（`hello-world`）で作業する。mainブランチが最新であることを確認してから開始する。

## 開始コマンド

```
/loop 30m
```

起動時に、そのターンで下記の「毎回のポーリング処理」を実行するよう指示する。例:

```
/loop 30m
Linearのhello-worldプロジェクトで自分にアサインされ、ステータスがTodoの課題を確認し、
あれば下記の手順書（_work/linear_loop_runbook.md）の「毎回のポーリング処理」に従って対応してください。
```

## 毎回のポーリング処理

1. Linear MCPの課題検索ツールで以下条件を検索する
   - プロジェクト = hello-world
   - 担当者 = 自分
   - ステータス = Todo
2. 該当0件 → 何もせず次回ポーリングまで待機。
3. 該当1件以上 → 更新日時が最も古い（＝先にアサインされた）課題を1件選ぶ。
4. 選んだ課題についてLinear側のステータスを「In Progress」に更新する。
5. 課題番号（例: `HEL-12`）から `feature/issue-HEL-12` の形式でブランチ名を機械的に生成し、`git checkout -b feature/issue-HEL-12` でブランチを作成する。
6. 課題のタイトル・説明文を要件として実装する。
   - 必要に応じて `_work/change_request.md` に対応するREQを追記してから着手する。
   - 実装方針・複数案の検討が必要な場合は `_work/change_plan.md` に案を追記する。
7. 実装後、`verify` スキル等で動作確認する。
8. 変更内容をユーザーに提示し、レビュー・承認を受ける。
   - 承認前にコミット・push・PR作成は行わない。
9. ユーザー承認後、そのブランチにコミットする。
   - push・PR作成はユーザーから明示的な指示があった場合のみ行う。
10. 完了したらLinear課題に以下をコメントする。
    - コミットハッシュ
    - 変更概要
11. Linear課題のステータスを「Done」または「レビュー待ち」に更新する。
12. 次回のポーリング（設定間隔後）まで待機し、1に戻る。

## ガードレール

- mainブランチへの直接コミット・pushは行わない。必ず `feature/issue-番号` ブランチ上で作業する。
- ユーザーレビューなしにコミットしない（案1の最大のメリットである「人による目視確認」を維持する）。
- Linear課題の内容が曖昧・矛盾している場合は実装を進めず、Linear課題へのコメントまたはユーザーへの確認を優先する。
- セッションを終了・スリープするとポーリングは止まる。再開する場合は本手順書の「開始コマンド」から再度 `/loop` を実行する。
