# 変更案（REQ1対応）

対象要望: [change_request.md](change_request.md) REQ1
「LINEARでタスク管理して、そこからclaude codeでローカルのこのプロジェクトに対して、割り振られたタスクをこなすように変更する」

対象Linearプロジェクト: https://linear.app/deepworkyk/project/hello-world-9b426137e05a/overview

## 前提

- Linear MCPコネクタ（claude.ai Linear）はclaude.ai側で認可済み。ただし本セッションの時点ではまだLinear関連ツールが認識されておらず、実際に利用する際はセッション再開等で認可情報が反映されているか確認すること。
- 利用可能になった際に呼び出せる想定の主なMCPツール: 課題検索（自分にアサインされ・指定ステータスの課題を取得）、課題コメント追加、課題ステータス更新。
- 本リポジトリは静的サイトのみで、CI/CD・package.json・自動化基盤は未整備（[CLAUDE.md](../CLAUDE.md) 参照）。

---

## 案1: 対話セッション + `/loop` によるポーリング運用（最小構成）

**概要**
ユーザーがローカルでClaude Codeセッションを開き、`/loop` スキルで一定間隔（例: 30分）ごとに「Linearの hello-world プロジェクトで自分にアサインされ、未着手（Todo）のIssueを確認し、あれば着手する」処理を回す。実装後は動作確認し、コミット前にユーザーが内容をレビューする。

**仕組み（具体手順）**
1. `/loop 30m` でポーリングセッションを開始する
2. 毎回のポーリングで、Linear MCPの課題検索ツールを使い「対象プロジェクト = hello-world」「担当者 = 自分」「ステータス = Todo」で課題を検索
3. 該当課題が0件なら何もせず次のポーリングまで待機。1件以上あれば、更新日時が古いもの（＝先にアサインされたもの）から1件選ぶ
4. 選んだ課題をLinear側で「In Progress」に更新し、課題番号（例: `HEL-12`）から `feature/issue-HEL-12` のようなブランチ名を機械的に生成して`git checkout -b`でブランチを作成
5. 作成したブランチ上でタイトル・説明文を要件として実装（必要に応じて`_work/change_request.md`にREQを追記してから着手）
6. 実装後は`verify`スキル等で動作確認し、変更内容をユーザーに提示。ユーザーが承認したらそのブランチにコミット（push・PR作成はユーザー指示があった場合のみ）
7. 完了したらLinear課題にコミットハッシュ/変更概要をコメントし、ステータスを「Done」または「レビュー待ち」に更新
8. 次のポーリング（30分後）まで待機し、2に戻る

**メリット**
- 追加インフラ不要、Claude Codeの標準機能のみで完結
- 人が都度目視確認できるため安全性が高い（コミット・push前にレビュー可能）
- 導入が最も速い

**デメリット**
- セッションを開いている間しか動かない（PCスリープ・終了で止まる）
- 「割り振られたら即実行」ではなく、ポーリング間隔分の遅延がある

## 案2: スケジュール実行（`schedule`スキル / cron）による半自動運用

**概要**
`schedule` スキルでcron登録した定期実行エージェント（例: 毎日朝9時）が、Linearの担当タスクを取得し、新規タスクがあればブランチを切って実装・検証し、コミットしてPull Requestを作成する。Linear側は「レビュー待ち」ステータスに更新し、PRリンクをコメントする。マージはユーザーが最終判断。

**仕組み**
1. cronトリガーでエージェントが起動
2. Linear MCPで自分担当・Todo/Readyステータスの課題を取得
3. 課題ごとに課題番号（例: `HEL-12`）から `feature/issue-HEL-12` のようなブランチ名を機械的に生成し、そのブランチを自動作成
4. 作成したブランチ上で実装 → 検証（`verify`スキル等）→ コミット
5. `gh pr create` でPR作成（ベースブランチ ← 作成した`feature/issue-番号`ブランチ）、LinearのIssueにPRリンクをコメントしステータス更新
6. mainへの直pushは行わず、必ずPRレビューを経由させる

**メリット**
- ローカルセッションを開き続ける必要がない、決まった時間に自動で回る
- PRベースなのでマージ前に人間が最終チェックできる

**デメリット**
- 無人実行のため、失敗時の検知・通知の仕組みが別途必要
- cron・PR運用のガードレール設計が必要（誤ったタスク解釈での実装リスク）
- 案1より設定・運用の手間が増える

## 案3: Linear Webhook起点のイベント駆動連携（フル自動）

**概要**
Linearでタスクが「対応中に割り当て」等の特定ステータス/ラベルに変わった瞬間にWebhookを発火させ、GitHub Actions等の受け口経由でClaude Codeをヘッドレス起動。Issue内容をそのままタスクとして実装し、ブランチ作成・実装・検証・PR作成・LinearへのPRリンク反映までを自動で行う。

**仕組み**
1. Linear側でWebhook設定（対象: 本プロジェクトのIssueステータス変更）
2. GitHub Actions（またはWebhook受信用の軽量サーバー）が発火を受け、Claude Code CLIを非対話モードで起動
3. 起動直後に、Webhookペイロードの課題番号（例: `HEL-12`）から `feature/issue-HEL-12` のようなブランチ名を機械的に生成し、そのブランチを自動作成
4. 作成したブランチ上でIssueのタイトル・説明をタスクとして実装 → 検証 → コミット → PR作成
5. Linear Issueに結果（PRリンク、失敗時はエラー内容）を自動コメント

**メリット**
- 「割り振られたら即座に着手」という要望に最も忠実
- 人がPCを操作していなくても動く、真の自動化

**デメリット**
- 実装コストが最も高い（Webhook受信基盤、CIシークレット管理、認証まわりの構築が必要）
- 現状CI未整備のリポジトリに新たな自動化基盤を追加することになり保守対象が増える
- 無人・即時実行のため誤動作時の影響範囲が一番大きく、ガードレール（PRのみ・自動マージ禁止・実行スコープ制限等）の設計が特に重要

---

## 所感

- まず案1で運用を試し、Linear連携のタスク粒度や指示の書き方に慣れてから案2・案3へ発展させるのが低リスク。
- 「即時着手」を重視するなら案3が要望に最も近いが、初期構築コストと安全対策の設計を要する。

# 変更案（REQ2対応）

対象要望: [change_request.md](change_request.md) REQ2
「下記の初回用のデータを作成する: index.html / css/style.css / js/app.js」

## 前提

- [CLAUDE.md](../CLAUDE.md) の想定構成では `js/app.js` が `data/gourmet.json` を読み込んで描画する設計になっており、「HTMLにコンテンツを直書きせず、データ（JSON）と描画ロジック（JS）を分離すること」と明記されている。
- そのためREQ2記載の3ファイルに加えて `data/gourmet.json` も初回データとして合わせて作成する（CLAUDE.mdの指示を満たすために必須）。
- 現状バックエンド・ビルドツールは一切なし。素のHTML/CSS/JSのみで完結させる。

## 設計方針

| ファイル | 役割 |
|---|---|
| `data/gourmet.json` | 名古屋グルメの初回サンプルデータ（数件） |
| `index.html` | ページ骨格。検索/絞り込みUIの器と、店舗一覧を描画するコンテナ（`<div id="gourmet-list">` 等）のみを持ち、店舗情報は書かない |
| `css/style.css` | 一覧のカード/グリッドレイアウトと最低限のレスポンシブ対応 |
| `js/app.js` | `fetch('data/gourmet.json')` でデータ取得し、DOM生成して `#gourmet-list` に描画 |

### `data/gourmet.json` のデータ構造案

```json
[
  {
    "id": "001",
    "name": "店名",
    "genre": "ラーメン",
    "area": "栄",
    "address": "愛知県名古屋市...",
    "description": "紹介文",
    "url": "https://example.com"
  }
]
```

初回は数件のサンプル（ダミーまたは実店舗数件）を入れ、実データの拡充は `research-json-generator` スキルで後追い対応する想定。

### `js/app.js` の処理フロー

1. `DOMContentLoaded` 後に `fetch('data/gourmet.json')`
2. 取得したJSON配列をもとにカード要素を生成し `#gourmet-list` に追加
3. 取得失敗時（JSON不備・パス誤りなど）はエラーメッセージを画面に表示

## 実装ステップ

1. `data/gourmet.json` を作成（サンプル数件）
2. `index.html` を作成（一覧表示用のコンテナと `css/style.css` / `js/app.js` の読み込みタグのみ）
3. `css/style.css` を作成
4. `js/app.js` を作成（fetch→描画、エラーハンドリング）
5. ローカルで動作確認する（`file://` 直開きだとブラウザによっては `fetch` がCORS制限で失敗するため、`python3 -m http.server` 等の簡易サーバー越しに確認する）
6. ユーザーレビュー後にコミット

## 留意点

- `file://` 直開き前提だと `fetch` が失敗する環境があるため、確認手順（簡易ローカルサーバー起動）を合わせて案内する。
- CLAUDE.mdの指示通り、店舗情報をHTMLへ直書きしない（すべて `data/gourmet.json` 経由）。
